DEPENDENCY STRENGTH SCORING (1-5 Scale)
════════════════════════════════════════════════════════════════════

**STRENGTH 5 — MANDATORY PREREQUISITE**
Rule B cannot be executed without Rule A being satisfied first.
AML Examples:
- CDD completion MUST precede account opening
- Customer risk rating MUST precede EDD determination
- Transaction monitoring alert MUST precede SAR investigation

**STRENGTH 4 — STRONG DEPENDENCY**
Rule B is heavily influenced or typically requires Rule A, though edge cases exist.
AML Examples:
- OFAC screening SHOULD precede wire transfer approval
- Beneficial ownership identification SHOULD precede CDD completion

**STRENGTH 3 — MODERATE DEPENDENCY**
Rule A provides context that meaningfully improves Rule B execution.
AML Examples:
- PEP screening results inform EDD scope
- Geographic risk factor informs product risk assessment

**STRENGTH 2 — WEAK DEPENDENCY**
Rule A and B are related but can be executed independently.
AML Examples:
- SAR narrative quality guidelines and SAR filing deadline rules
- Annual CDD refresh timing and periodic review scope

**STRENGTH 1 — INFORMATIONAL LINK**
Rules reference the same topic but have no execution dependency.
AML Examples:
- Two different regulatory citations covering same CTR threshold
- Parallel FATF and FinCEN rules about cash structuring prohibition

4. **Regulatory Alignment (10% weight):**
   - 100: Dependency consistent across BSA, FinCEN, and FATF
   - 70: Consistent in primary framework (BSA/FinCEN)
   - 50: Framework-specific dependency
   - 30: Potential conflict between frameworks

DEPENDENCY TYPE DEFINITIONS
════════════════════════════════════════════════════════════════════

Use EXACTLY one of these seven lowercase values for `dependency_type`:

1. **prerequisite** — Rule A MUST be satisfied before Rule B can be evaluated/executed
   Characteristics:
   - Temporal ordering is mandatory
   - B cannot proceed until A is satisfied
   - Violation of A makes B irrelevant or impossible
   - Typical Strength: 4-5 (STRONG to CRITICAL)

   AML Examples:
   - Identity verification → CDD completion (Strength: 5 — account cannot open without it)
   - Customer risk rating → EDD determination (Strength: 5 — EDD threshold cannot be checked without a risk tier)
   - Transaction monitoring alert → SAR investigation (Strength: 5 — investigation cannot start without an alert)
   - CDD completion → Beneficial ownership certification (Strength: 4 — BO collection is part of CDD close-out)

2. **sequential** — Rules that must execute in a specific order for business process flow
   Characteristics:
   - Order matters for workflow efficiency and regulatory timeline
   - Not strictly dependent, but logically ordered
   - Common in multi-step filing and review processes
   - Typical Strength: 3-4 (MODERATE to STRONG)

   AML Examples:
   - CTR chain: Cash detection → Aggregation check → CTR preparation → CTR filing within 15 days (Strength: 4)
   - SAR chain: Alert generation → Alert review → Escalation → Investigation → SAR submission within 30 days (Strength: 4)
   - OFAC chain: Data collection → SDN match check → Block decision → Blocked funds report (Strength: 3)
   - Periodic review chain: Risk tier assignment → Review scheduling → CDD refresh → Updated risk rating (Strength: 3)

3. **conditional** — Rule B only applies when a specific condition in Rule A is met
   Characteristics:
   - B is triggered by A's outcome or threshold breach
   - If A's condition is not met, B does not apply
   - Common in threshold-based AML triggers
   - Typical Strength: 3-5 (depends on regulatory mandate)

   AML Examples:
   - IF cash transaction ≥ $10,000 THEN CTR required (Strength: 5 — BSA mandate)
   - IF customer risk = HIGH THEN EDD required (Strength: 5 — CDD Rule 31 CFR §1010.230)
   - IF SAR filed THEN tipping-off prohibition applies (Strength: 5 — 31 U.S.C. §5318(g)(2))
   - IF OFAC SDN match confirmed THEN transaction blocking required (Strength: 5)
   - IF PEP identified THEN enhanced screening required (Strength: 4)

4. **complementary** — Rules work together to achieve a comprehensive compliance goal
   Characteristics:
   - Multiple rules must all be satisfied
   - Each addresses a different aspect of the same requirement
   - Combined validation provides complete compliance
   - Typical Strength: 3-4 (MODERATE to STRONG)

   AML Examples:
   - KYC four pillars: identity verification + address verification + source of funds + beneficial ownership — all required for CDD (Strength: 4)
   - Three-screen CDD: OFAC screening + PEP screening + adverse media — all must clear before account approval (Strength: 4)
   - SAR filing package: narrative quality + supporting transaction data + supervisor sign-off — all required (Strength: 3)
   - AML program pillars: internal controls + independent audit + designated BSA officer + training — all required under BSA (Strength: 4)

5. **contradictory** — Rules that conflict and require reconciliation
   Characteristics:
   - Rules have opposing or incompatible requirements
   - May be context-dependent (different jurisdictions or frameworks)
   - Requires escalation or policy decision
   - Typical Strength: 3-4 (MODERATE to STRONG — must resolve)

   AML Examples:
   - FinCEN CTR threshold ($10,000) vs some state-level reporting thresholds (Strength: 4)
   - FATF Recommendation 16 wire transfer requirements vs FinCEN Travel Rule thresholds (Strength: 4)
   - EU AMLD5 beneficial ownership transparency vs BSA confidentiality provisions (Strength: 4)
   - Same-day SAR filing for terrorist transactions vs 30-day standard SAR deadline (Strength: 3)

6. **override** — Rule B supersedes or modifies Rule A under specific conditions
   Characteristics:
   - Exception or special-case handling
   - B explicitly replaces or narrows A's requirement
   - Often requires elevated approval or regulatory license
   - Typical Strength: 4-5 (STRONG to CRITICAL)

   AML Examples:
   - OFAC specific license overrides wire transfer blocking requirement (Strength: 5)
   - Safe harbor provision (31 U.S.C. §5318(g)(3)) overrides disclosure prohibition for SAR sharing with affiliates (Strength: 5)
   - Law enforcement request overrides standard SAR non-disclosure to customer (Strength: 5)
   - Supervisory exception overrides automated alert closure threshold (Strength: 4)

7. **validation** — Rule B verifies or validates the output or requirement of Rule A
   Characteristics:
   - B checks A's result for accuracy, completeness, or compliance
   - Quality-control or second-level review relationship
   - Common in AML sign-off and audit chains
   - Typical Strength: 3-4 (MODERATE to STRONG)

   AML Examples:
   - Second-level SAR narrative review validates initial analyst draft (Strength: 4)
   - Supervisor sign-off validates OFAC false-positive determination (Strength: 4)
   - Compliance audit validates completeness of CDD files (Strength: 4)
   - Quality assurance review validates CTR accuracy before submission (Strength: 3)

════════════════════════════════════════════════════════════════════
CIRCULAR DEPENDENCY DETECTION
════════════════════════════════════════════════════════════════════

**Definition**: A circular dependency exists when Rule A depends on Rule B, and Rule B depends on Rule A (directly or through intermediary rules), creating an unresolvable compliance loop.

**Detection Rules:**
1. **Direct Circular**: A → B → A
2. **Indirect Circular**: A → B → C → A
3. **Conditional Circular**: A conditionally depends on B, B depends on A

**AML-Specific Risk:**
- Circular dependencies in SAR/CTR chains create regulatory deadline ambiguity
- Circular screening rules may inadvertently allow prohibited transactions
- Circular CDD dependencies can stall account opening indefinitely

**Resolution Strategies:**
1. **Refactor Rules**: Split circular rules into non-circular components
2. **Add Entry Point**: Identify which rule initiates the cycle
3. **Flag for Manual Review**: Escalate to BSA Officer for clarification
4. **Context Disambiguation**: Clarify under which framework each rule applies

**Output Requirements:**
- List all circular dependency chains found
- Describe the loop: "BR001 → BR045 → BR089 → BR001"
- Explain the compliance ambiguity
- Suggest resolution strategy
- Mark severity: HIGH (direct), MEDIUM (indirect), LOW (conditional)

════════════════════════════════════════════════════════════════════
CONFLICT DETECTION
════════════════════════════════════════════════════════════════════

Flag `contradictory` dependencies when:
- Two rules assign different thresholds to the same metric (e.g., FinCEN vs state-level SAR threshold)
- Rules from different frameworks contradict each other (FATF Recommendation vs FinCEN rule)
- A safe harbor or override rule contradicts a general requirement from another rule
- EU AMLD provisions conflict with BSA provisions for the same customer activity

Flag circular dependencies separately — they are not the same as `contradictory` type.
