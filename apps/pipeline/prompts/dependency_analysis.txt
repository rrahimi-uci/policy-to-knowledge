════════════════════════════════════════════════════════════════════
STRENGTH SCALE (1-5)
════════════════════════════════════════════════════════════════════

**5 - CRITICAL (Absolute Blocker)**
- Target rule CANNOT execute without source rule satisfied
- Workflow completely stops if dependency violated
- Legal/regulatory mandate (not discretionary)
- Examples:
  * Borrower authorization → Credit pull (legal requirement)
  * Property appraisal → LTV calculation (mathematical dependency)
  * Income verification → DTI calculation (data dependency)

**4 - STRONG (Major Impact)**
- Target rule severely compromised without source
- High risk of compliance failure or process breakdown
- Standard business practice (not optional)
- Examples:
  * Initial disclosure → Application acceptance (TRID timeline)
  * Document collection → Underwriting start (process efficiency)
  * Credit check → Rate lock (pricing accuracy)

**3 - MODERATE (Important Relationship)**
- Target rule affected but may have workarounds
- Affects efficiency, accuracy, or risk level
- Best practice but not absolute requirement
- Examples:
  * Pre-qualification → Property search (borrower benefit)
  * Title search → Closing preparation (risk mitigation)
  * Secondary income verification → Primary verification (quality control)

**2 - WEAK (Advisory/Informational)**
- Loose relationship, helpful but not critical
- Affects optimization or user experience
- Contextual relevance
- Examples:
  * Historical loan data → Current underwriting (reference)
  * Similar property comps → Subject appraisal (market context)
  * Industry best practices → Lender policy (guidance)

**1 - MINIMAL (Tangential)**
- Very loose association
- May share entity or domain but independent logic
- Informational only
- Examples:
  * General disclosure rule → Specific calculation rule (same domain)
  * Different loan product rules (same borrower)

**DEPENDENCY THRESHOLD: Only document dependencies rated ≥ 3 (MODERATE or higher)**

════════════════════════════════════════════════════════════════════
CIRCULAR DEPENDENCY DETECTION
════════════════════════════════════════════════════════════════════

**Definition**: A circular dependency exists when Rule A depends on Rule B, and Rule B depends on Rule A (directly or through intermediary rules), creating an unresolvable loop.

**Detection Rules:**
1. **Direct Circular**: A → B → A
2. **Indirect Circular**: A → B → C → A
3. **Conditional Circular**: A conditionally depends on B, B depends on A

**Impact of Circular Dependencies:**
- ❌ Workflow deadlock (cannot determine execution order)
- ❌ Infinite loops in automated systems
- ❌ Ambiguous compliance requirements
- ❌ Cannot determine critical path

**Resolution Strategies:**
1. **Refactor Rules**: Split circular rules into non-circular components
2. **Add Entry Point**: Identify which rule initiates the cycle
3. **Flag for Manual Review**: Escalate to compliance team for clarification
4. **Context Disambiguation**: Clarify under which contexts each rule applies

**Output Requirements:**
- List all circular dependency chains found
- Describe the loop: "BR001 → BR045 → BR089 → BR001"
- Explain the conflict/ambiguity
- Suggest resolution strategy
- Mark severity: HIGH (direct), MEDIUM (indirect), LOW (conditional)

════════════════════════════════════════════════════════════════════
DEPENDENCY ANALYSIS FRAMEWORK
════════════════════════════════════════════════════════════════════

1. **prerequisite** - Rule A MUST be satisfied before Rule B can be evaluated/executed
   Characteristics:
   - Temporal ordering is mandatory
   - B cannot proceed until A is satisfied
   - Violation of A makes B irrelevant or impossible
   - Typical Strength: 4-5 (STRONG to CRITICAL)
   
   Examples:
   - Borrower authorization → Credit document retrieval (Strength: 5)
   - Property appraisal → LTV calculation (Strength: 5)
   - Income verification → DTI calculation (Strength: 5)
   - Loan application submission → Underwriting rules (Strength: 4)

2. **sequential** - Rules that must execute in specific order for business process flow
   Characteristics:
   - Order matters for workflow efficiency
   - Not strictly dependent, but logically sequential
   - Common in multi-step processes
   - Typical Strength: 3-4 (MODERATE to STRONG)
   
   Examples:
   - Initial disclosure (TRID) → Application → Processing → Underwriting → Closing (Strength: 4)
   - Document collection → Review → Verification → Approval (Strength: 3)
   - Appraisal order → Inspection → Report → Review (Strength: 3)

3. **conditional** - Rule B only applies when specific condition in Rule A is met
   Characteristics:
   - B is triggered by A's outcome
   - If A is false/not met, B doesn't apply
   - Common in branching logic
   - Typical Strength: 3-5 (depends on criticality)
   
   Examples:
   - IF LTV > 80% THEN PMI required (Strength: 5 - regulatory mandate)
   - IF investment property THEN reserves required (Strength: 4 - standard requirement)
   - IF manufactured home THEN HUD certification required (Strength: 5)
   - IF subordinate financing THEN additional approval required (Strength: 4)

4. **complementary** - Rules work together to achieve a comprehensive compliance goal
   Characteristics:
   - Multiple rules must all be satisfied
   - Each addresses different aspect of same requirement
   - Combined validation provides complete compliance
   - Typical Strength: 3-4 (MODERATE to STRONG)
   
   Examples:
   - Multiple documentation rules for borrower identity verification (Strength: 4)
   - Various property eligibility rules for same loan type (Strength: 3)
   - Set of underwriting requirements for specific loan program (Strength: 4)
   - Combined rules for fraud prevention (Strength: 4)

5. **contradictory** - Rules that may conflict and require reconciliation
   Characteristics:
   - Rules appear to have opposing requirements
   - May be context-dependent (different programs/jurisdictions)
   - Requires escalation or policy decision
   - Typical Strength: 3-4 (MODERATE to STRONG - must resolve)
   
   Examples:
   - Federal vs state disclosure requirements with different timeframes (Strength: 4)
   - Conflicting LTV limits between investor overlays and agency guidelines (Strength: 4)
   - Different credit score requirements for same loan type (Strength: 3)

6. **override** - Rule B supersedes or modifies Rule A under specific conditions
   Characteristics:
   - Exception or special case handling
   - B explicitly replaces or modifies A's requirement
   - Often manual intervention or elevated approval
   - Typical Strength: 4-5 (STRONG to CRITICAL)
   
   Examples:
   - Manual underwriting overrides automated DU/LP decisions (Strength: 5)
   - Executive exception overrides standard approval limits (Strength: 5)
   - State law overrides federal guideline when more restrictive (Strength: 5)
   - Grace period rules override standard late payment penalties (Strength: 4)

7. **validation** - Rule B verifies or validates the output/requirement of Rule A
   Characteristics:
   - B checks A's result for accuracy/compliance
   - Quality control relationship
   - Common in multi-layer validation
   - Typical Strength: 3-4 (MODERATE to STRONG)
   
   Examples:
   - QC review validates initial appraisal (Strength: 3)
   - Secondary income verification validates primary income docs (Strength: 4)
   - Compliance audit validates loan file completeness (Strength: 4)
